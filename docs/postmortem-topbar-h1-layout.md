# Postmortem: выравнивание заголовка страницы под топбар «Баланс»

Версия: 2026-05-26  
Статус: **задача закрыта откатом** — pixel-perfect выравнивание не достигнуто; в прод восстановлена базовая схема до экспериментов.  
Контекст: Streamlit 1.57, `layout="wide"`, приложение `/opt/sinlex/app.py`, shell `/opt/sinlex/page_shell.py`.

Связанные документы:

- [`TZ-flow-tokens-payment.md`](TZ-flow-tokens-payment.md) — §0 (что **осталось** в проде по топбару и пополнению)
- [`page_shell.py`](../page_shell.py) — `render_app_flow_balance_bar()`, `page_title()`
- [`app.py`](../app.py) — `_inject_main_scroll_clamp()`, глобальные стили main

История правок в чате Cursor: transcript `621dd1ad-7a02-42dc-9beb-6772148b43a5` (май 2026).

---

## 1. Исходная цель

После внедрения глобального топбара «Баланс» (`position: fixed`) заказчик хотел:

1. Главный заголовок каждой страницы (`page_title()` / `h1`) — **сразу под нижним краем** топбара, без видимого зазора.
2. Прокрутка main **не уходила выше** заголовка (нельзя «подтянуть» контент так, чтобы заголовок оказался под полосой «Баланс»).
3. На странице «Библиотека инструмента» (`page == "materials"`) дополнительно поднять блок контента (~100 px) — как эталон «правильного» уровня заголовка.

Топбар и пополнение «Поток» при этом должны были **продолжать работать** (см. §0 в ТЗ по токенам).

---

## 2. Что уже работало до экспериментов с заголовком

К моменту начала работ над выравниванием (ориентир — правки **после** фиксации fixed-топбара, **до** серии патчей «контент строго под топбаром») в проде было:

| Компонент | Назначение |
|-----------|------------|
| CSS `position: fixed` на `[class*="st-key-app_topbar_flow"]` | полоса «Баланс» не скроллится |
| JS `fixTopbar()` в `_inject_main_scroll_clamp()` | `left`/`width` от `stMain`, `--sinlex-topbar-h`, синхронизация spacer |
| `#sinlex-topbar-spacer` | отступ в потоке под высоту топбара |
| `<div class="sinlex-page-top">` в `page_title()` | якорь верхней границы прокрутки |
| `st.html(..., unsafe_allow_javascript=True)` | без этого скрипт в Streamlit 1.57 не выполняется |

Этого хватало, чтобы топбар был виден и контент не заезжал **под** бар. Требование «0 px между баром и `h1`» оказалось **значительно сложнее**.

## 3. Хронология неудачных подходов

Ниже — сжатая хронология; многие приёмы применялись **одновременно**, что усугубляло отладку.

### 3.1. Двойная компенсация высоты топбара

- **`padding-top: var(--sinlex-topbar-h)`** на `[data-testid="stMainBlockContainer"]` / `block-container` (`_APP_MAIN_LAYOUT_CSS` в `page_shell.py`).
- Параллельно — **spacer** `#sinlex-topbar-spacer` и **`minScroll`** с учётом `topbarHeight()`.

**Симптом:** контент уезжал вниз, внизу экрана пустая зона; на части страниц зазор у топбара не менялся.

**Причина:** один и тот же зазор задавался дважды; плюс в `app.py` глобально стояло `padding-top: 0 !important`, которое **перебивало** отступы из shell.

### 3.2. Глобальный якорь `#sinlex-content-start`

- Введены `render_main_content_start()` и div **выше** контента страниц.
- Якорь прокрутки перенесён с `.sinlex-page-top` / `h1` на `sinlex-content-start` или spacer.

**Симптом:** «заголовок связан с топбаром» — визуально заголовок не совпадал с линией, до которой clamp’ится скролл.

**Причина:** две разные «верхние линии» (низ топбара vs якорь прокрутки).

### 3.3. Схлопывание обёртки топбара (`height: 0`)

```css
[data-testid="stMain"] [data-testid="stElementContainer"]:has(> [class*="st-key-app_topbar_flow"]) {
    height: 0 !important;
    ...
}
```

**Симптом:** прокручивается ~¼ высоты экрана; под контентом большое белое поле.

**Причина:** в Streamlit родительский `stVerticalBlock` часто оборачивает **не только** топбар, а весь столбец main — схлопнулся весь scrollable area.

### 3.4. `transform: translateY(...)` на main

Попытка поднять блок страниц (в т.ч. `materials: -100px`).

**Симптом:** топбар исчезал при скролле или вёл себя как обычный блок.

**Причина:** у предка с `transform` элемент с `position: fixed` позиционируется относительно этого предка, а не viewport.

### 3.5. JS `fixGap()` — отрицательный `margin-top`

Измерение расстояния от низа топбара до `h1` (~144 px в debug) и подтягивание `#sinlex-content-start` или `h1`.

**Симптом:** «ничего не поменялось» или сдвиг на другой странице ломал вёрстку.

**Причины:**

- rerun Streamlit пересобирает DOM — inline-стили сбрасываются или применяются к другому узлу;
- зазор складывался из нескольких margins/paddings — одним margin закрывался не тот слой;
- параметр `extra_pull_px` для `materials` вставлялся через f-string в Python и давал **SyntaxError** — код не деплоился, хотя `systemctl` мог показывать `active`.

### 3.6. Агрессивный CSS для `h1` / `stTitle` / `stHeading`

Обнуление `margin-top` / `padding-top` у всех заголовков в main.

**Симптом:** нестабильно между страницами (`page_modules` vs inline в `app.py`); конфликт с отступом `0.35rem` на `block-container`.

### 3.7. Побочные правки

- `forceSidebarOpen()` в скрипте прокрутки — лечение симптома сворачивания сайдбара, не заголовка.
- Debug-badge (`topbarH`, `gap`) — полезен, но не устранил корень.
- Скрытие кнопки сворачивания сайдбара и `initial_sidebar_state="expanded"` — **отдельная** тема; при откате заголовка часть оставлена намеренно.

## 4. Наблюдаемые симптомы и их объяснение

| Жалоба пользователя | Вероятная техническая причина |
|---------------------|-------------------------------|
| «Ничего не поменялось» | Конфликт `padding-top: 0 !important` vs spacer; JS не выполнялся; SyntaxError в `app.py` |
| «Всё съехало вниз» | Spacer + `padding-top` на container + `minScroll` с `topbarHeight()` |
| «Прокрутка только четверть экрана» | `height: 0` на родителе main / `stVerticalBlock` |
| «Топбар пропал / едет» | `transform` на предке fixed-элемента |
| «Заголовок под топбаром при скролле» | Якорь прокрутки не совпадает с визуальным `h1`; двойной ghost-height fixed в потоке |
| «На Библиотеке −100 px не работает» | Разная DOM-структура страницы; сломанный деплой скрипта |

## 5. Откат (2026-05-26)

По запросу заказчика откатили **все изменения темы выравнивания заголовка**, вернув схему **до** блока правок «контент строго под топбаром» (ориентир — состояние до ~строки 866 в transcript).

### Убрано

- `render_main_content_start()` / `#sinlex-content-start`
- `fixGap`, `extra_pull_px`, `forceSidebarOpen` в скрипте
- CSS схлопывания `stElementContainer` с `:has(topbar)`
- `transform`, лишние правила для `stTitle` / `stHeading`
- класс `sinlex-has-topbar`, spacer на `var(--sinlex-topbar-h)` в CSS (в shell снова **3rem** в статике; JS по-прежнему может подстраивать spacer)

### Восстановлено

- `page_title()`: снова `<div class="sinlex-page-top"></div>` перед `st.title()`
- Якорь JS: `.sinlex-page-top` || `h1`
- `minScroll = anchorTop(...) - topbarHeight() - 2`
- `padding-top: 0.35rem` на `section[data-testid="stMain"] > div.block-container`
- Порядок после сайдбара: `_inject_main_scroll_clamp()` → глобальный CSS → `_render_app_flow_balance_bar()`

### Оставлено (не часть отката)

- Fixed-топбар + `fixTopbar()` + popover «Баланс»
- Развёрнутый сайдбар (CSS + `initial_sidebar_state="expanded"`)

## 6. Корневые причины провала (выводы)

1. **Streamlit не даёт стабильной «линии начала контента».** Отступы размазаны по `stMainBlockContainer`, `block-container`, цепочкам `stElementContainer` / `stVerticalBlock`, пустым markdown-узлам.

2. **Fixed + in-flow ghost.** `position: fixed` убирает элемент из визуального потока, но обёртка виджета в DOM часто **сохраняет высоту** — spacer и collapse пытались компенсировать одно и то же разными способами.

3. **Два контура стилей.** `app.py` и `page_shell.py` с конкурирующими `!important` — правки в одном файле нивелировались другим.

4. **Смешение целей.** «Не скроллить выше заголовка» и «заголовок в 0 px от топбара» — связаны, но решаются **разными** якорями и отступами; подмена якоря на `content-start` ухудшила UX.

5. **Процесс.** Много гипотез без заморозки baseline и без запрета опасных приёмов (`height: 0` на `:has(main)`, `transform` на main).

6. **Окружение.** `st.html` без `unsafe_allow_javascript`; ошибки вставки JS в Python; отсутствие единого чеклиста «скрипт реально выполнился» на 2–3 страницах.

## 7. Рекомендации, если задачу возобновят

Не повторять откатившиеся патчи. Возможный безопасный порядок:

1. **Диагностика на 3 страницах** (Dashboard, Projects, materials): в DevTools или временном overlay — кто даёт N px между `getBoundingClientRect()` низа топбара и верха `h1` (padding/margin какого `data-testid`).

2. **Один рычаг отступа под топбар:**
   - либо только spacer (высота = `fixTopbar()` → `h`),
   - либо только `padding-top` на **одном** выбранном контейнере,
   - не оба сразу.

3. **Якорь прокрутки** — только `.sinlex-page-top` или первый `h1` страницы, не глобальный div выше топбара.

4. **Запретить:** `height: 0` / `max-height: 0` на предках `stMain`; `transform` на `stMainBlockContainer` и выше; отрицательные margin без измерения на каждой странице.

5. **Стили топбара/main** — один файл (или один блок `st.markdown` в `app.py`), без дублирующих `padding-top: 0 !important`.

6. **Критерий готовности:** визуальный зазор ≤ 2 px и нет регресса высоты scroll area на всех пунктах меню.

## 9. Краткая формулировка для заказчика

Мы не смогли надёжно прижать заголовки к топбару, потому что интерфейс собран Streamlit-виджетами с непредсказуемыми отступами, а попытки «подтянуть» вёрстку несколькими способами сразу ломали прокрутку и сам топбар. Откат вернул стабильную работу приложения; идеальное выравнивание потребует отдельной задачи с диагностикой по страницам и **одного** согласованного механизма отступа.
